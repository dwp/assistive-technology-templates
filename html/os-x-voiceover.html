About this document

Voiceover should always be used with the Safari browser. Using it with any other browser may cause unexpected behaviour.

This Voiceover test is designed to test whether your website or service is likely to work using common functions of the Voiceover screenreader. It will not tell you if your service will work for every user, but it will help you get an understanding as to how well you can expect your service to work for Voiceover users.

You should also test your service with real users of Voiceover and other assistive technologies.

Read all page content

Confirm that the content on the page can be read and that the navigation and page content follow a logical reading order.

How to check: Read all page content

  1. Press CTRL + ALT + FN + Left Arrow to go to the top of the page
  2. Press CTRL + ALT + A to read all content from current position
  3. Listen to all of the page content

What to check: Read all page content

Confirm that:

  • all content is read out correctly
  • all content follows a logical reading order
  • the context is consistent between what is visible and what is read out
  • content is not repeated; unless the repeated content is intentional in the design
  • icons are read out in a meaningful way
  • images have meaningful alternative text
  • content does not rely on sensory characteristics, for example: "On the left"

Landmarks present

Confirm the page has semantic landmarks.

How to check: Landmarks present

  1. Press CTRL + ALT + U to display the "Rotor"
  2. Press Right Arrow to get to "Landmarks"
  3. Check each landmark in the list

What to check: Landmarks present

Confirm that:

  • landmarks make sense and their role is accurate.
  • the page has a header landmark (banner).
  • the page has a footer landmark (contentinfo).
  • the page has a main landmark.
  • any aside landmarks are not contained inside another top level landmark such as the main.
  • any additional header and footers do not have the role of banner and contentinfo. For example an article header.
  • regions have not been incorrectly used to contain content that should otherwise be inside a named landmark.

Dynamic content

Pop-ups, modals and expanded sections should be announced. Anything which causes the page content to change without refreshing must be available to screenreader users.

How to check: Dynamic content

  1. Activate the element which triggers the dynamic content to change
  2. Wait for the screen reader to finish talking
  3. Observe whether the changes were announced

What to check: Dynamic content

Confirm that:

  • when content changed dynamically the screenreader notifies you of changes
  • the announcement is polite, and does not interrupt the screenreader unless it is an emergency

Validation errors

When a form cannot be submitted due to validation errors, those errors should be announced to a screenreader. When a validation error appears on the page, make sure it is the next thing which is announced and make sure that it makes sense.

How to check: Validation errors

  1. Fill in a form component incorrectly
  2. Progress until the validation error appears
  3. Once the error loads, wait for the screen reader to stop talking and observe whether it announced that there was an error
  4. Press CTRL + ALT + SHIFT + I to see if the title announces an error
  5. Press CTRL + ALT + CMD + J to move through the form fields and see if it obvious which field caused the error
  6. Observe whether the field which caused the error explains how to fix it

What to check: Validation errors

Confirm that:

  • when validation errors occur, the screenreader announced to the user that there is an error
  • the title of the page changes to indicate there is an error, for example: Error: Your details
  • it is clear from the screenreader which field caused the error and how to fix it

Page title is descriptive

The page title should describe the current page, the website name and if it is a sub-site it should contain the parent site. For example:
What is your name? - Apply for Universal Credit - GOV.UK

With the exception of a home page, the page title should match the H1. For example:
h1: What is your name?
title: What is your name? - Apply for Universal Credit - GOV.UK

If the H1 contains personal information, then the H1 should match as closely as possible. If the service is using analytics software, the page title will be sent and stored, so personally identifiable information should never be used in the page title. For example:
h1: What is John Smith's date of birth?
title: What is your partners date of birth?

How to check: Page title is descriptive

  1. Press CTRL + ALT + SHIFT + I to read out the page title

What to check: Page title is descriptive

Confirm that:

  • the page title describes the page
  • the page title describes the website
  • the page title does not contain personally identifiable information
  • the page title and the h1 match (unless personally identifiable)

Links make sense out of context

Every link should be predictable, even if the surrounding context is removed. No links should just say 'Read more' or 'click here'.

How to check: Links make sense out of context

  1. Press CTRL + ALT + U to open the Element list
  2. Navigate to the links list
  3. Check every link

What to check: Links make sense out of context

Confirm that:

  • all links make sense out of context
  • all links work and their destination is expected
  • no links have the same description but different destinations

No automatic change of context

Unless there is a clear warning that the behaviour of an element has been modified, an element should not cause a change of context unless they receive deliberate input. For example: a form should not submit when you select a checkbox.

A change in context means the content you were reading or the task you were busy doing has changed. For example, clicking a link and changing the page.

How to check: No automatic change of context

  1. Press CTRL + ALT + FN + Left Arrow to go back to the top of the page
  2. Press tab to navigate through the interactive elements
  3. Press Arrow Keys to navigate through radios and checkboxes

What to check: No automatic change of context

Confirm that one of the following is true:

  • there is no change in context without using a link or a button
  • there is a clear warning for any modifications which cause a change of context

Bypass blocks

Content which is repeated on multiple pages should have a mechanism which allows the user to skip past it. For example: a skip to main content link.

How to check: Bypass blocks

  1. Press CTRL + ALT + FN + Left Arrow to go to the top of the page
  2. Press tab and look for an element which skips over repetitive content such as navigation menus

What to check: Bypass blocks

Confirm that:

  • a skip link is present that links to the main content
  • the skip link is the first interactive element on the page
  • any other blocks of repetitive content are also skipped or have their own skip links

Navigate by landmarks

A user should be able to navigate the page by landmarks. For example, you should be able to jump between the header to the main content to the footer without having to go through all of the elements contained in each of those sections.

How to check: Navigate by landmarks

  1. Press CTRL + ALT + FN + Left Arrow to go back to the top of the page
  2. Press CTRL + ALT + SHIFT + Right Arrow to move through each landmark

What to check: Navigate by landmarks

Confirm that:

  • you can navigate to every landmark on the page
  • it is obvious which landmark is selected
  • all landmarks are selected in a logical order
  • all important content is contained within a landmark

Navigate by tables

A user should be able to navigate the page by tables.

How to check: Navigate by tables

  1. Press CTRL + ALT + FN + Left Arrow to navigate to the top of the page
  2. Press CTRL + ALT + CMD + T to move through each of the tables on the page

  • no hidden tables can be selected
